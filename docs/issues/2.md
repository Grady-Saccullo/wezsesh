## TL;DR — verdicts

| # | Assumption (as written in design.md) | Verdict | Why |
|---|---|---|---|
| A1 | §13.4 — "resurrect uses `io.open` and ignores POSIX advisory locks" | **CONFIRMED** | Only file primitive in the write path is `io.open(path, "w+")`; no `flock` / `fcntl` / `F_SETLK` / `lockf` anywhere in the plugin. (`file_io.lua:13`, harness `00`.) |
| A2 | §13.4 — Phase B reply: `state_manager.save_state` returns / errors so the binary can map ok=false | **REFUTED** (with one caveat) | `state_manager.save_state` returns `nil` for I/O and encryption errors — these are swallowed by an internal `pcall` and surfaced **only** via `wezterm.emit("resurrect.error", "Failed to write state: …")`. **Caveat:** `wezterm.json_encode(state)` is called *before* the inner pcall, so `save_state` CAN raise via json_encode failures — V4a confirmed `json_encode` throws on non-encodable values. (`file_io.lua:74,78–93`, `state_manager.lua:26–34`, harness `01`, real-wezterm V6.) |
| A3 | §9.4.2 — `pcall(resurrect.state_manager.save_state, current_state)` is the error gate for `SAVE_FAILED` | **REFUTED (dual-path)** | The pcall is necessary but not sufficient. It catches `json_encode` failures (real-wezterm V4a: `throw` with `error converting Lua function to JsonValue`) but never sees I/O / encryption errors (real-wezterm V6: `pcall_ok=true, ret=nil, resurrect.error fired`). The Lua handler must consult **both** the pcall return AND a `resurrect.error` capture buffer. |
| A4 | Appendix C — `resurrect.workspace_state.restore_workspace.finished` "only fires on the success path and cannot be used as a completion signal" | **CONFIRMED** | `workspace_state.lua:14` emits `…start`; line 45 emits `…finished` only after the `for window_states` loop completes without raising. `wezterm.mux.spawn_window` and `restore_window` are not pcall-protected, so any mid-loop error skips the `…finished` emit. |
| A5 | Appendix C — `resurrect.error` is the right hook for save-failure observability | **CONFIRMED** (with caveats) | Every save/load/encrypt/decrypt error path inside resurrect emits `resurrect.error` with a single `string` arg. It is the only fire-and-forget signal of failure. Caveat: there is **no correlation id** — handlers must correlate by timing/path against the active save. |
| A6 | §13.6 — encryption providers: `age` / `rage` / `gpg` | **CONFIRMED** | `encryption.lua:69–84` builds the encrypt argv. For `gpg` the command differs (`--batch --yes --encrypt --recipient … --output …`); for `age`/`rage` it's `<method> -r <pubkey> -o <path>`. Decrypt mirrors with `-d -i <privkey>` (age/rage) or `--batch --yes --decrypt` (gpg). (Harness `04`.) |
| A7 | §13.6 — `state_manager.load_state` decrypts encrypted snapshots | **CONFIRMED** with footnote | Decryption goes through `file_io.load_json` → `pub.encryption.decrypt(file_path)`. On wrong-key/decrypt-failure the error is again **swallowed** into `resurrect.error`; `load_state` returns `{}`. Same swallowing problem as A2/A3 applies to `SNAPSHOT_LOAD_FAILED`. |

Three of seven assumptions are **REFUTED** (A2, A3) or need amendment scoping. §13.4's *locking* premise is intact; §13.4's *error-detection contract* is broken. §9.4.2 needs a rewrite around a dual-path detector (pcall return + `resurrect.error` capture). Appendix C is correct but undercommunicates the consequences for the save path. New behavioral spec for `SAVE_FAILED` is provided in §F below; runtime confirmation in §H.

## A. `state_manager.save_state` — file-write code path

```
wezsesh save handler
   │
   └── resurrect.state_manager.save_state(state, opt_name?)               state_manager.lua:26
         │  -- dispatches by shape: window_states→workspace, tabs→window,
         │     pane_tree→tab. For wezsesh, always the "workspace" arm.
         │
         └── file_io.write_state(file_path, state, "workspace")           file_io.lua:72
               wezterm.emit("resurrect.file_io.write_state.start", …)    line 73
               json_state = wezterm.json_encode(state)                    line 74
               json_state = sanitize_json(json_state)                     line 75
                 wezterm.emit("resurrect.file_io.sanitize_json.start")
                 -- gsub "[\x00-\x1F]" → "\\u00XX"
                 wezterm.emit("resurrect.file_io.sanitize_json.finished")
               if pub.encryption.enable then                              line 76
                  emit "resurrect.file_io.encrypt.start"
                  ok, err = pcall(pub.encryption.encrypt(file_path,
                                                         json_state))
                  if not ok:
                    emit "resurrect.error", "Encryption failed: "..err    line 82
                    log_error(…)                                          line 83
                  else:
                    emit "resurrect.file_io.encrypt.finished"             line 85
               else:                                                      line 87
                  ok, err = pub.write_file(file_path, json_state)
                                                                          file_io.lua:11
                    └── pcall(function()                                  line 12
                          local handle = io.open(file_path, "w+")         line 13   ← truncate-and-write
                          if not handle then error("Could not open …") end
                          handle:write(str)                               line 17
                          handle:flush()                                  line 18
                          handle:close()                                  line 19
                       end) → returns ok, err
                  if not ok:
                    emit "resurrect.error", "Failed to write state: "..err line 90
                    log_error(…)                                          line 91
               wezterm.emit("resurrect.file_io.write_state.finished", …) line 94
                                                                            ↑
                                                                   ALWAYS fires —
                                                                   not a success
                                                                   signal.
```

**Properties of the write:**

* `io.open(path, "w+")` truncates the existing file at open and writes in place. **Not** atomic (no temp+rename). Crash mid-write → on-disk file is shorter than intended. Concurrent cross-process writers race; last `close()` wins (harness `02`).
* No `fsync`. After `close()` the bytes are buffered in the kernel page cache; a host crash within the writeback window can lose the write entirely, leaving stale prior bytes.
* No advisory lock. wezsesh's `safefs.AcquireExclusive` lock is invisible to resurrect; there is **no** mutual exclusion between wezsesh's brief Phase A/Phase C locks and resurrect's `io.open`.

These three properties together are why the §13.4 premise stands: even a held POSIX advisory lock would not delay resurrect's writer, so wezsesh holding the lock across IPC would buy nothing.

## B. `state_manager.save_state` error contract

* **Return value:** `nil`. Always.
* **Lua error:** none. The internal `pcall` in `file_io.write_file` (line 12) catches every error from `io.open` / `write` / `flush` / `close`. The encryption arm has its own `pcall` (line 78) that catches `error("Encryption failed: …")` raised by `encryption.lua:83`. Neither pcall rethrows.
* **Side-effect signal:** `wezterm.emit("resurrect.error", string)` fires with one of these messages:

  | Origin | Lua line | Message shape |
  |---|---|---|
  | Plaintext write — `io.open` failed | `file_io.lua:90` | `"Failed to write state: …/file_io.lua:15: Could not open file: <path>"` |
  | Plaintext write — `write`/`flush`/`close` failed | `file_io.lua:90` | `"Failed to write state: <runtime-error-string>"` |
  | Encryption — `<method>` exit nonzero / process spawn failure | `file_io.lua:82` | `"Encryption failed: <stderr or 'Failed to execute: <cmd>'>"` |
  | Encryption — wrong key / agent missing (decrypt) | `file_io.lua:107` | `"Decryption failed: <stderr>"` |
  | `delete_state` — `os.remove` returned false | `state_manager.lua:136` | `"Failed to delete state: <path>"` |
  | `load_state` — `json_parse` returned nil | `state_manager.lua:44` | `"Invalid json: <path>"` |

**All payloads are bare strings, not tables.** A correlation id, status-code, file-path tuple, etc. would have to be parsed back out of the human message string, which is brittle. We treat the entire string as `details.raw_error` and avoid pattern-matching on it for control flow (only for the user-visible toast).

* **Empirical verification:** `01_swallowed_error.lua` (run successfully on this machine, lua-5.4.7):

  ```
  pcall ok    = true
  return value= nil
  emit  resurrect.file_io.write_state.start  …/spike.json  workspace
  emit  resurrect.file_io.sanitize_json.start  {"workspace":"spike",…}
  emit  resurrect.file_io.sanitize_json.finished
  emit  resurrect.error  Failed to write state: …/file_io.lua:15: Could not open file: …
  emit  resurrect.file_io.write_state.finished  …/spike.json  workspace
  ```

## D. Encryption providers and round-trip behavior

* **Default:** `age` (`encryption.lua:9`).
* **Supported `method` strings:** any binary on PATH; the only special-cased branch is `string.find(method, "gpg")`. So `age`, `rage`, `age2`, `gpg`, `gpg2` all work — for non-`gpg` the argv is `<method> -r <pubkey> -o <path>` with stdin = json (encrypt) and `<method> -d -i <privkey> <path>` capturing stdout (decrypt). For names containing "gpg" the argv switches to gpg's batch flags.
* **Encrypt stdin transport size threshold (encryption.lua:30):** on POSIX hosts, payloads under 150,000 bytes are passed via `echo -E -n <input> | <cmd>` (a constructed shell pipeline). Larger payloads go through `io.popen(cmd, "w")` directly. Both paths exist; both are exercised in steady-state. Wezsesh-relevant: a workspace with restore_text on many panes can easily exceed 150 KB → both code paths must work.
* **Decrypt:** always `wezterm.run_child_process({…})` (encryption.lua:96). On nonzero exit it `error()`s with `"Decryption failed: " .. stderr`, which is caught by the outer pcall in `file_io.load_json` and surfaced as a `resurrect.error` event.
* **Magic-byte sniff (Appendix B) is correct for age:** harness `04` shows the first 22 bytes of an age-encrypted file are `age-encryption.org/v1\n`.

`gpg`, `rage`, `age` were the brief's required coverage. `rage` and `gpg` were not exercised here (out of scope per the issue body, "age + gpg sample is sufficient; rage / openpgp variants out") but follow the same shape by inspection.

## F. Amendment proposals

### F.1 — `docs/design.md §13.4` save-flow reshape

Phase B as currently written assumes Lua replies `completed` `ok=false` on
error. With A2 refuted, that path is impossible without an additional
mechanism. Proposed Phase B (replacement text):

```
PHASE B — emit save OSC (no lock held):
dispatcher.Dispatch(ipcCtx, "save", { name, overwrite, expected_hash })
  Lua handler:
    1. Drain the per-pane resurrect.error queue (clear leftovers).
    2. pcall(resurrect.state_manager.save_state, state, opt_name)
       ⇒ pcall ALWAYS returns true, nil for save_state per A2.
    3. Inspect the per-pane queue for resurrect.error events emitted
       between (1) and (2):
         • if any: result.reply_error(payload, "SAVE_FAILED",
             err_string, { raw_error = err_string }) and return.
         • if none: result.reply_completed(payload, { name = name }).
  Reply mapping (binary-side):
    "completed" ok=true       → proceed to Phase C
    "completed" ok=false      → propagate SAVE_FAILED to TUI; return
                                (Lua includes details.raw_error)
```

Implementation hooks the queue via a single module-level
`wezterm.on("resurrect.error", function(msg) … end)` listener registered at
plugin load. The listener appends `(timestamp, msg)` to a per-pane FIFO
keyed off the pane that owns the in-flight save. (`wezterm.emit` runs
handlers synchronously on the same thread that called the emitter, so the
listener observes the error before `state_manager.save_state` returns.)

The §13.4 *caveat 2.b* paragraph that motivates the lock-briefly design
("resurrect uses io.open and ignores POSIX locks") still stands and is
re-affirmed by A1 — no edit needed there.

### F.2 — `docs/design.md §9.4.2` Save handler rewrite

Replace the existing four bullets with:

```
1. Receive payload. Lua does NOT enforce expected_hash — checked binary-side.
2. Snapshot-and-clear the resurrect.error queue for this pane (a module-
   level listener funnels all resurrect.error emissions into a per-pane
   FIFO; see §9.5/§9.6).
3. pcall(resurrect.state_manager.save_state, current_state).
   Per spike #2: this pcall is always (true, nil) — the call succeeds on
   the Lua side regardless of disk I/O outcome. Treat it as "we tried".
4. Read back the resurrect.error queue.
   • Empty → result.reply_completed(payload, { name = name }).
   • Non-empty → result.reply_error(payload, "SAVE_FAILED",
       table.concat(messages, "\n"),
       { raw_error = table.concat(messages, "\n") }).
5. The binary fills `hash` after re-reading the file under a brief second
   lock (§13.4 step 7).
```

### F.3 — `docs/design.md §9.4.1` Restore handler — clarify pcall contract

The existing §9.4.1 already wraps `restore_workspace` in pcall on the wezsesh
side, which is correct because resurrect itself does not pcall its own
spawn loop (workspace_state.lua:32–43). Add one sentence to §9.4.1:

> Resurrect's `restore_workspace` does not internally pcall its window/tab
> spawn loop, so the wezsesh-side pcall is load-bearing — without it,
> a single bad cwd in the snapshot terminates the Lua handler before
> `result.reply_*` can fire. (Confirmed by spike #2.)

Also: on a `RESURRECT_PARTIAL` pcall catch, drain the per-pane
`resurrect.error` queue and append its contents to the warning's
`details.raw_error` for parity with `SAVE_FAILED`.

### F.4 — Appendix C — expand the events table

Replace the three-line table with:

```
| Event | Handler | Purpose |
|---|---|---|
| `resurrect.file_io.write_state.start`    | state.set_writing(path, true)       | snapshot-write gate (entry) |
| `resurrect.file_io.write_state.finished` | state.set_writing(path, nil)        | snapshot-write gate (exit; fires on success AND error) |
| `resurrect.error`                        | per-pane FIFO + log_warn            | save / load / decrypt / encrypt failure observability |

Notes:
  • write_state.finished is NOT a success signal — it fires unconditionally
    after the encode + write attempt. resurrect.error MUST be consulted to
    distinguish success from failure.
  • All restore_*.finished events fire only on the success path (resurrect
    does not pcall its restore loops). wezsesh wraps restore calls in pcall
    on the Lua handler side and surfaces RESURRECT_PARTIAL on caught errors.
  • resurrect.error is fire-and-forget with a single string arg; there is
    no correlation id. wezsesh correlates by routing through a per-pane
    FIFO and draining around each save / load call (see §9.4.2).
```

### F.5 — `docs/design.md §7` — `SAVE_FAILED` behavioral spec

Append to the existing `SAVE_FAILED` row:

> `details.raw_error` is the unparsed string emitted by
> `wezterm.emit("resurrect.error", …)` from inside resurrect. wezsesh does
> NOT pattern-match on the message for control flow — every distinct write
> failure (ENOSPC, EACCES, EROFS, encryption-agent missing, encryption-agent
> nonzero exit) yields a `SAVE_FAILED` with a different opaque string. The
> TUI toast surfaces the first 256 bytes of `raw_error` verbatim (sanitised
> per §15.4).

The known message-prefix taxonomy (for the toast copy table only — never for
control flow) is:

| Prefix | Origin | Probable cause |
|---|---|---|
| `Failed to write state: …Could not open file:` | `io.open(path, "w+")` returned nil | parent dir gone, EACCES, EROFS |
| `Failed to write state: …` (other) | `write` / `flush` / `close` raised | ENOSPC, EIO, broken pipe |
| `Encryption failed: …Failed to execute:` | `io.popen(cmd, "w")` returned nil | encryption binary missing on PATH |
| `Encryption failed: …` (other) | encryption process exit nonzero or stderr non-empty | wrong recipient, agent error |
| `Decryption failed: …` | decryption process exit nonzero | wrong key, file corrupted, agent error |

These are observational — adding new prefixes does not break wezsesh. Used
only by `internal/tui/strings.go` to pick a "disk full?" / "agent missing?"
helper line under the verbatim raw_error.

### F.6 — `docs/design.md §13.6` — encrypted snapshot ops table

The §13.6 row "save (overwrite) | yes | hash over raw bytes; resurrect rewrites"
remains correct, but add a footnote referencing F.1 — encryption errors are
indistinguishable in shape from disk-write errors at the wezsesh-binary
level; both surface as `SAVE_FAILED` with a `raw_error` string. The split is
done only by the TUI helper line (F.5).

## G. Out-of-scope items observed during the audit (not addressed here)

These are noted for triage but do **not** block the spike's acceptance
criteria.

* `resurrect.state_manager.save_state(state, opt_name)` does **not**
  validate `opt_name`. Names containing `/` are rewritten to `+` (matching
  wezsesh's §12.3 transform), but other characters (`.`, `..`, leading `-`,
  control bytes) pass through. wezsesh validates names binary-side per §15
  before the OSC, so this is safe **as long as the binary owns name
  validation**. Doctor should warn if a foreign tool wrote a snapshot file
  with a name that fails §15.
* `state_manager.load_state` swallows `wezterm.json_parse` failures *only*
  when json_parse returns nil; if it `error()`s instead, the error
  propagates to the caller. **Resolved by §H V3:** on this wezterm build
  `wezterm.json_parse` raises (serde_json `EOF while parsing a value`) —
  so torn JSON propagates through `load_state` to the caller's pcall,
  bypassing resurrect's `if not json then` arm and the `Invalid json`
  emit. `SNAPSHOT_LOAD_FAILED.details.raw_error` carries the serde_json
  error string verbatim.
* `periodic_save` re-arms via `wezterm.time.call_after`, which runs on the
  Lua event loop. Within a single wezterm process, periodic and wezsesh
  saves serialise. Across wezterm processes targeting the same snapshot
  dir, they race per A1 — wezsesh's `safefs.AcquireExclusive` Phase A/C
  windows narrow but do not eliminate the race. This is the documented
  "concurrent non-wezsesh writes clobber freely" v0.1 limitation in §13.4.

## H. Real-wezterm validation results

Five probes were run against `wezterm 0-unstable-2026-03-31`
(aarch64-apple-darwin, mlua/Lua 5.4) on 2026-04-30 via the on-demand
harness at `plugin/spike2/probes.lua`, invoked from the wezterm Debug
Overlay (`p.run_safe()`). Raw output is in
`~/.cache/wezsesh-spike/resurrect_audit_validation.txt`. Verdicts
below; full transcript reproduced in §H.6.

| Probe | Verdict | Confirms / refines |
|---|---|---|
| **V1** wezterm.emit synchrony | **PASS** | Trace = `before_register, before_emit, handler_fired, after_emit`. `wezterm.on` handlers run inline before `wezterm.emit` returns. § F.1 / §9.13 *per-call* capture is sufficient — no per-pane FIFO needed. |
| **V2** multiple `wezterm.on` handlers | **PASS** | 3 handlers fired in registration order (`h1,h2,h3`). §F.4 / §9.1 wiring can coexist with user-installed `resurrect.error` handlers without shadowing. |
| **V3** `wezterm.json_parse` on torn JSON | **throw** | Error: `EOF while parsing a value at line 1 column 5` (serde_json formatted). Confirms `SNAPSHOT_LOAD_FAILED` arrives via the caller's pcall — NOT via resurrect's `Invalid json` emit (which only fires when json_parse returns nil). §F.5 raw_error taxonomy adds an entry: serde_json error strings. |
| **V4a** `wezterm.json_encode` on function value | **throw** | Error: `error converting Lua function to value (while processing "f": error converting Lua function to JsonValue)`. **Confirms the spike's self-correction:** `state_manager.save_state` is *not* universally non-raising — `json_encode(state)` is called before resurrect's inner pcall (file_io.lua:74), so cycles / function values / non-encodable userdata in the workspace state DO propagate through `pcall(save_state, …)`. §9.4.2 must be a dual-path detector (pcall return + capture buffer). |
| **V6** `state_manager.save_state` swallows I/O errors | **PASS** | `pcall_ok=true`, `return=nil`, `resurrect.error` fired with `Failed to write state: …Could not open file: …`, `write_state.finished` fired afterward. **Headline finding empirically confirmed against real wezterm**, not just the Lua-stub harness. ### H.1 Decision impact summary

* `with_capture` in §F.1 / §9.13 stays *per-call* (V1 PASS).
* `wezsesh.resurrect_error` listener installs alongside any user listener
  without conflict (V2 PASS).
* `SNAPSHOT_LOAD_FAILED` `details.raw_error` shape is now pinned: serde_json
  formatted strings (V3 throw). §F.5 message-prefix table adds:
  `EOF while parsing a value` / `expected value` / `trailing characters` etc.
* §9.4.2 save handler is **dual-path**: pcall(save_state) catches
  json_encode failures (V4a throw); a `with_capture` capture catches I/O
  and encryption failures (V6 PASS). Both must be inspected before
  replying success. Pseudocode:

  ```
  local pok, perr, captured = with_capture(function()
    return resurrect.state_manager.save_state(state)
  end)
  if not pok then
    return reply_error(payload, "SAVE_FAILED",
                       tostring(perr), { raw_error = tostring(perr) })
  end
  if #captured > 0 then
    local raw = table.concat(captured, " | ")
    return reply_error(payload, "SAVE_FAILED",
                       raw, { raw_error = raw })
  end
  return reply_completed(payload, { name = name })
  ```

### H.2 Probes intentionally not run in real wezterm

* **V5** `restore_workspace.finished` only on success path — already proven
  by source inspection (`workspace_state.lua:9–46` has no internal pcall
  around the spawn loop; the `…finished` emit at line 45 is unreachable
  when the loop body raises). Earlier auto-running version of this probe
  locked up wezterm because it tried to interact with the mux on the GUI
  thread during config eval.
* **V7** synthetic `resurrect.error` emit — redundant with V1 + V2, and
  fan-out to the user's tabline.wez `"resurrect"` extension during config
  eval was a contributing factor in the lockup.
* **V4b** cyclic-table `json_encode` — stack-overflow risk in the C
  serializer. V4a (function value) covers the same design question.
