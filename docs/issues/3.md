# Spike #3 — wezterm Lua version + OSC/render interleaving

**Status:** complete (Lua-version probe + OSC interleave repro v2 on darwin-arm64).
**Wezterm tested:** `wezterm 0-unstable-2026-03-31` build hash `577474d`,
target triple `aarch64-apple-darwin`, mlua/Lua 5.4.
**Spec sections under review:** `docs/design.md` §3.1 (OSC atomicity rationale),
§3.5 (4 KiB ceiling), §9.0 (Lua-version requirement), §9.0.1 (mlua sandbox),
§9.9 (`ct_eq.lua`), §17.6 (e2e smoke test), §8.17.1 (`wezterm.lua_version`).
**Coverage caveat:** darwin-arm64 only. Linux runs not yet performed —
recommendations on the linux-amd64 / linux-arm64 §8.17.1 floor are
extrapolated from the wezterm release matrix and need a manual rerun.

Three of nine premises are **REFUTED** (A4, A5, A6); the rest amend.
The spike's "need to amend §3.1 / §3.5" headline is upgraded by v2 from

## B. Probe B — OSC interleave (v2 sweep, three modes)

**Run protocol.** Three back-to-back runs in a fresh native wezterm
pane (no tmux/screen, no Claude-Code agent PTY); listener installed
via `loadfile` from `wezterm.lua`. Each run sweeps a 15-bucket payload
list with **100 emits per bucket** at 50 ms cadence (1500 emits / run,
~75 s). Per-run summaries written to `~/.cache/wezsesh-spike/osc_sent.txt`,
joined by `run_id` against `~/.cache/wezsesh-spike/osc_received.txt` via
`docs/issues/3-harness/analyze.sh`.

| Mode | Bubbletea output | OSC writer | Mutex shared? |
|---|---|---|---|
| `render` | `os.Stdout` | `/dev/tty` | NO — reproduces §3.1 status quo |
| `render-mutex` | `tea.WithOutput(sharedTTY)` (mutex-locked io.Writer) | `/dev/tty` (same mutex) | YES — proposed mitigation |
| `no-render` | (no TUI) | `/dev/tty` | n/a — pure baseline |

**Reads.**

* No loss at JSON ≤ 1280 B (on-wire ≤ 1734 B). Loss appears at JSON
  1536 B (on-wire 2074 B) and persists at every larger bucket, but
  with high variance.
* Loss is **non-monotonic** with size (4 % at 2304 B, 3 % at 2560 B,
  but 0 % at 2048 B and 3840 B). This is the signature of a
  *probabilistic* race — every emit has some chance of byte-level
  interleave with a bubbletea frame write at the kernel TTY layer,
  and that chance grows once the OSC needs more than one
  `write(2)` syscall to drain.
* Consistent with the §3.1 model: the kernel TTY buffer on darwin
  is small enough (~1 KiB cited in §3.1) that OSCs whose total wire
  bytes exceed ~1.5–2 KiB are split into multiple syscalls. Between
  splits, bubbletea's frame paint can interleave from a different fd.
  When wezterm's parser sees an embedded `ESC[…` from bubbletea
  mid-OSC, it cancels the OSC per ECMA-48 and routes the trailing
  bytes to the rendered text stream — visible as base64 leakage.
* Visual confirmation of the leakage: see v1 transcript in
  `docs/issues/3-harness/README.md` (the `hYWFhYWFhY…` artifact is
  base64 of the `aaaa…` filler section).

**Reads.**

* **0 % loss across all 1500 emits, including the 6170 B on-wire bucket.**
* Implementation: bubbletea's renderer was constructed with
  `tea.WithOutput(sharedTTY)`, where `sharedTTY` is an `io.Writer`
  that takes the same `sync.Mutex` as the OSC writer's `emit()` path

  bubbletea's frame writes against OSC emits at the user-space level;
  the kernel can still split each individual write across multiple
  syscalls, but the two streams never interleave with each other.
* **The §3.1 mitigation is empirically correct.**

  during the back-to-back invocations, a single rendered frame caught
  the transition between two runs and showed `frame=4401` (run 1's
  final paint) and `frame=1` (run 2's first paint) on the same row.
  This is a bubbletea inline-renderer artifact at a *program* boundary
  (the second invocation began before wezterm finished scrolling the
  first), not a mutex issue. Within-run renders were clean and OSC
  delivery was 100 %.

**Reads.**

* **0 % loss across all 1500 emits and 0 orphan files** at run-end.
* `on_wire_B` in the table reports the size the OSC *would have been*
  under inline (Alt A) transport for that JSON bucket. The actual OSC
  on the wire is constant: ~150–180 B for every emit (the pointer JSON
  varies only in `seq` length). The "headline" payloads up to 4608 B /
  6170 B on-wire under Alt A are routed through the file in this mode.
* **Mechanism.** Each `emit()` does `O_EXCL` create of
  `~/.cache/wezsesh-spike/sidecar/<run_id>/<run_id>-<seq>.json` (mode
  0600) → write JSON → `fsync` → `rename(tmp, final)` → emit a tiny OSC
  carrying `{"sidecar":"<path>","run_id":"<id>","seq":N}`. The
  `user-var-changed` listener decodes the pointer, opens the file,
  reads the full payload, deletes the file, and records the same
  bucket / seq / decoded byte length the inline path would have.
* **Latency budget.** ~5 ms p50 / ~8 ms p95. Two orders of magnitude
  slower than the inline OSC write (which is sub-100 μs in `mode=
  render-mutex`) but invisible at wezsesh's verb cadence — verbs fire
  on user picker actions, not at sustained 60 Hz. p95 of 7.5 ms
  approximates one frame at 120 fps.
* **Failure-mode visibility.** A dropped OSC under Alt E does not
  silently vanish — the corresponding sidecar file remains undeleted
  in the runtime dir. The Go binary counts orphans at run-end as a
  free diagnostic. Production wezsesh would surface this via a doctor
  check (`runtime.dir.req_orphans`).

Alt E (sidecar).** §C.0 below records the architecture decision.

## C. Proposed amendments to `docs/design.md`

These are draft text and locator notes only — actual edits land via
the `docs/spike-3-amendments` PR after this findings doc is reviewed.

### C.0 — Architecture decision: Alt E (sidecar) over Alt A (mutex)

Both mitigations are validated at 0 % loss across the same 1500-emit
sweep up to 6170 B on-wire (§B.2 vs §B.4). Choosing Alt E for v1:

| Property | Alt A (mutex) | Alt E (sidecar) |
|---|---|---|
| OSC race | eliminated by mutex | structurally impossible (OSC ≤ 256 B) |
| Per-emit latency | sub-100 μs | ~5 ms p50 |
| Cross-package coupling | `internal/tui` and `internal/uservar` share a writer | none — independent transports |
| Defense-in-depth ceiling | needed (§3.5 → 1 KiB JSON) | not needed for correctness |
| Plugin complexity | none | file read + delete + path validation |
| Failure-mode visibility | latent (mutex violations are silent) | visible (orphan files = lost OSCs) |
| Disk footprint | none | one transient JSON per verb |
| Doctor checks added | none | one (`runtime.dir.req_orphans`) |

**Decision rationale.** Alt E's coupling cost is genuinely zero — the
TUI doesn't share state with the verb-emission path; bubbletea is
free-running. Failure modes are visible on disk rather than latent in
a contract reader-of-tea-source-code can violate by accident. The
~5 ms emit latency is invisible at wezsesh's verb cadence (one OSC
per user picker action). The plugin gains a small file-IO step the
existing `safefs.AtomicWriteFile` discipline already supports
(snapshot writes, sidecar writes, state.json — same atomic
tmp+rename pattern).

The remaining sub-sections describe the spec edits this implies.

### C.1 — §3.1 — Forward path rewrite (file-resident request +
        pointer OSC)

Replace the section's body (lines from "Bytes written to `/dev/tty`…"
through the atomicity-rationale paragraph and the `internal/uservar.Writer.mu`
mutex-and-bubbletea sentence) with:

> **Two-phase forward dispatch.** The binary first writes the
> canonical-JSON request bytes (§3.3) to a per-request file under
> `<runtime_dir>/req/`, then emits a fixed-shape OSC carrying only a
> pointer to that file:
>
> 1. **Atomic file write.** `safefs.AtomicWriteFile` —
>    `O_WRONLY|O_CREAT|O_EXCL` on `<runtime_dir>/req/<8-hex>.json.tmp`,
>    write canonical-JSON request bytes, `fsync`, `rename` to
>    `<runtime_dir>/req/<8-hex>.json`. Mode 0600. `<8-hex>` is the
>    first 8 hex chars of the request ULID (same prefix as §3.2's
>    reply socket — both share the request's identity). Symlink
>    defense via `safefs.Enforce(SymlinkRefuse)` on every ancestor.
> 2. **Pointer OSC.** Bytes written to `/dev/tty` (NOT `os.Stdout`),
>    under `internal/uservar.Writer.mu`:
>
>    ```
>    ESC ] 1337 ; SetUserVar=wezsesh_op= <base64-pointer> BEL
>    0x1B 0x5D "1337;SetUserVar=wezsesh_op=" <b64> 0x07
>    ```
>
>    `<b64>` is `base64.StdEncoding` of the canonical-JSON pointer:
>
>    ```jsonc
>    {
>      "v": 1,
>      "id":   "<26-char Crockford-base32 ULID>",
>      "path": "<absolute path to the request file>"
>    }
>    ```
>
>    Total on-the-wire OSC stays ≤ 256 B (§3.5). The pointer carries
>    its own `id`/`v` so the plugin can reject a malformed pointer
>    without ever opening the file.
>
> **Atomicity rationale.** The on-the-wire OSC at ≤ 256 B fits in a
> single kernel `write(2)` syscall on every supported platform
> (darwin TTY ≥ 1 KiB, Linux ≥ 4 KiB), so the renderer-OSC interleave
> race spike #3 documented for inline-payload OSCs is **structurally
> impossible**: the kernel TTY layer serialises whole syscalls.
> The full request remains arbitrary-sized in the file, where it is
> never subject to the OSC parser's byte-stream concerns.
>
> **Plugin handler outline.** `user-var-changed` for `wezsesh_op`:
>   1. `b64.decode` + `wezterm.json_parse` → pointer.
>   2. Pointer-shape validate (§9.3.1): `{v: number == 1, id:
>      string len 26, path: string starting with the configured
>      `runtime_dir/req/`}`. Rejects symlink-traversal attempts
>      and any path outside the prefix.
>   3. `io.open(path, "r")` → file contents (canonical-JSON request).
>      Mode-stat check: must be regular file, mode 0600, owner-self.
>   4. `os.remove(path)` (best-effort; orphans handled by §C.5
>      doctor sweep).
>   5. `wezterm.json_parse` of file contents → request payload.
>   6. Hand off to existing §9.3 steps (a)–(i) which run unchanged
>      against the file-derived payload.
>
> The `internal/uservar.Writer.mu` mutex still serialises wezsesh's
> own pointer-OSC writes against any concurrent OSC the binary itself
> may emit. It does not need to coordinate with bubbletea's renderer
> any more — the pointer OSC's single-syscall property removes that
> requirement.

(Keep the existing **PTY-multiplexer caveat** paragraph verbatim —
it applies identically to the pointer OSC.)

### C.6 — §8.5 / §8.6 / §8.8 — Dispatcher + uservar package APIs

**§8.5 (`internal/ipc.Dispatcher` interface):** add a `ReqDir` field
to the dispatcher dependencies. The `Dispatch(...)` body now does:

```go
// Pseudo-code; actual signatures per §8.5/§8.6.
func (d *dispatcher) Dispatch(ctx, verb, args) (chan, error) {
    payload := canonical.Encode(buildRequest(verb, args))
    if err := safefs.AtomicWriteFile(
        filepath.Join(d.ReqDir, ulidPrefix+".json"),
        payload, 0o600,
    ); err != nil { return nil, err }
    pointer := canonical.Encode(map[string]any{
        "v":    1,
        "id":   ulid,
        "path": filepath.Join(d.ReqDir, ulidPrefix+".json"),
    })
    if err := d.uv.WriteOSC(ctx, base64.StdEncoding.EncodeToString(pointer)); err != nil {
        return nil, err
    }
    return d.replies(ulid), nil
}
```

**§8.8 (`internal/uservar.Writer`):** revise the docstring to match
the new contract — the bytes the writer emits are the pointer-OSC
payload, never the full request:

> // WriteOSC emits one OSC 1337 SetUserVar=wezsesh_op=<payload>
> // sequence. payload MUST be base64'd canonical-JSON of the §3.1
> // pointer (≤ ~250 B). Single write(2) syscall on every supported
> // platform — the §3.1 atomicity argument relies on this. Callers
> // MUST NOT use this path for >256 B payloads; sidecar-write the

### C.9 — §9.0.1.1 (new) — Lua expression-call ambiguity

> **Lua expression-call ambiguity.** Lua does NOT insert statement
> separators across newlines when the next line starts with `(`. So
> ```lua
> wezsesh.setup({ binary = REPO .. "/wezsesh" })
> (loadfile(REPO .. "/some/probe.lua"))()
> ```
> parses as a *single* chained call: `setup({...})(loadfile(...))()`.
> The `setup` return value is then implicitly invoked as a function
> and raises "attempt to call a nil value". Plugin code MUST either
> capture the chunk into a local first (`local probe = loadfile(...) probe()`)
> or prefix the call with a leading `;`. CI lint SHOULD flag any line
> matching `^\s*\(` immediately following an expression-call
> statement.

### C.10 — §9.3 / §9.3.1 — Plugin handler step pipeline

Insert four new steps **before** the existing (a)–(i) — they convert
the OSC pointer into the file-derived payload that (a)–(i) operate on
unchanged:

```lua
-- (pre-1) Pointer JSON parse (pcall). Reject non-JSON / non-table.
-- (pre-2) Pointer field-shape validator (§9.3.1.A below).
-- (pre-3) Path-prefix + mode + non-symlink + io.open guard.
-- (pre-4) os.remove(path) best-effort, then wezterm.json_parse(file).
--         Result becomes `payload` for the existing (a)–(i) flow.
```

§9.3.1 grows two validators:

```lua
-- §9.3.1.A — pointer shape (NEW)
return type(pointer.v) == "number" and pointer.v == 1
   and type(pointer.id) == "string" and #pointer.id == 26
   and type(pointer.path) == "string"
        and pointer.path:sub(1, #req_dir_prefix) == req_dir_prefix

-- §9.3.1.B — payload shape (existing §9.3.1, renamed for clarity)
return type(payload.v) == "number" and payload.v == 1
   and type(payload.id) == "string" and #payload.id == 26
   and type(payload.ts) == "number"
   and type(payload.op) == "string" and #payload.op > 0 and #payload.op <= 32
   and type(payload.args) == "table"
   and type(payload.reply_sock) == "string"
        and #payload.reply_sock > 0 and #payload.reply_sock <= 104
   and type(payload.target_window_id) == "number"
   and type(payload.hmac) == "string" and #payload.hmac == 64
```

Note: `pointer.id` MUST equal `payload.id` (§9.3 step pre-4) — a
mismatch indicates either a stale on-disk file or a forged pointer.
Rejection is wire-silent; emits `REQ_POINTER_REJECTED` to log.
