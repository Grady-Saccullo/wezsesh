## Manual test runbook

The spike plugin writes its generated HMAC key to `~/.cache/wezsesh-spike/key` (mode 0600) and a structured events log to `~/.cache/wezsesh-spike/events.log`. The runner script `scripts/spike-test.sh` orchestrates scenarios 4a/4b/4c against your real wezterm and grep-asserts the events log; you no longer need to copy the key out of the debug overlay.

**Prereq:** you are in the wezsesh repo, inside the nix dev-shell (`nix develop`), and have built the binary (`go build -o ./wezsesh ./cmd/wezsesh`).

> **Important — must be a native wezterm pane.** The spike emits its OSC to `/dev/tty`, so any PTY multiplexer or agent that owns the shell's PTY (tmux, screen, Claude Code, asciinema, etc.) will consume the OSC bytes as program output and they will never reach wezterm's parser. `scripts/spike-test.sh` refuses to run if it detects one; if the test fails with empty events log even from a fresh tab, run `scripts/spike-diag.sh` to see what owns your PTY.

### Step 1 — Add the spike plugin to your wezterm config

Edit your `~/.wezterm.lua` (or `${XDG_CONFIG_HOME}/wezterm/wezterm.lua`) and add:

```lua
local wezterm = require 'wezterm'
local config = wezterm.config_builder and wezterm.config_builder() or {}

-- Spike loader (replace REPO with the absolute path to your wezsesh checkout)
local REPO = '/abs/path/to/wezsesh'
-- Bust the require cache so disk edits to wezsesh.* submodules land on
-- each Ctrl+Shift+R reload (without this, only init.lua re-runs and the
-- cached submodules from the first load stay active).
for k in pairs(package.loaded) do
    if k:sub(1, 8) == 'wezsesh.' then
        package.loaded[k] = nil
    end
end
local wezsesh = (loadfile(REPO .. '/plugin/init.lua'))()
wezsesh.setup({ binary = REPO .. '/wezsesh' })

return config
```

Restart wezterm (or `Ctrl+Shift+R` to reload). Each reload regenerates the events log and rewrites the keyfile, so your shell's exported `WEZSESH_HMAC_KEY` will be stale until you re-source `scripts/spike-env.sh`.

### Step 2 — Confirm the plugin loaded

Open the wezterm debug overlay (default keybind: `Ctrl+Shift+L`). You should see:

```
wezsesh spike: registered. key+events under /Users/<you>/.cache/wezsesh-spike, binary=/.../wezsesh
```

Also confirm the artefacts on disk:

```sh
ls -la ~/.cache/wezsesh-spike/
# expect: key (mode 0600, 65 bytes), events.log (0 bytes initially)
```

If you instead see `wezsesh: no source of randomness; HMAC disabled` in the debug overlay, your binary path is wrong or `/dev/urandom` is unreadable.

## Findings

### Round-trip latency

Single-run observation: **8.3 ms** (binary stdout: `OK reply received in 8.252833ms (8.3 ms)`). `scripts/spike-test.sh` reports 5/5 successful runs. The 100 ms target is comfortably met for the trivial-payload `noop` verb on a quiescent darwin/arm64 machine; restore-class verbs (`switch` / `load`) which call into resurrect will be the real latency stress test, deferred to their own tickets.

### Replay drop

`scripts/spike-test.sh --replay` passes:
- First OSC: `RESULT result=ok id=01KQGN8M…`
- Second OSC (same id): `RESULT result=replay id=01KQGN8M…`

The 2 s retransmit path (`tea.Tick` in §13.1) also gets dedupe-dropped — observed accidentally during a state-bug repro where the binary timed out on the first request but the retransmit landed and produced `RESULT result=replay`. Confirms the §5.4 `seen_ids` gate covers both explicit duplicates and the harmless retransmit.

### HMAC mismatch

`scripts/spike-test.sh --corrupt-hmac` passes. Events log shows `HMAC_MISMATCH id=…` and never a `RESULT result=…`; the binary exits with code 3 (`IPC_TIMEOUT — no reply within 5s`). Wire-silent semantics (§7) confirmed.

### Byte-equality at runtime

Go's `sans-hmac.sha256=69c0c3ceaf…` matched Lua's `verifier canonical-bytes-sha256=69c0c3ceaf…` to the bit during the happy-path run, in addition to the unit-test fixture proof. The `args` empty-object tagging (§4.2) round-trips correctly through `wezterm.json_parse` → field-shape validate → `tag_noop` → re-encode.

## Recommendation: proceed with proposed amendments

The wire model in §3 / §4 / §5 / §6.5 / §9.3 is sound: round-trip latency is well under target, HMAC verify + replay drop work as specified, and the canonical-JSON encoder is byte-symmetric across Go and Lua. **No reshape required.** Five concrete amendments to land before the implementation backlog opens:

1. **mlua sandbox lacks `debug`** (§9.x): plugin code MUST NOT use `debug.getinfo`, `debug.traceback`, or any other debug-library function. Auto-locating the plugin root via `debug.getinfo(1, "S").source` does not work; spec the entry-point as taking an explicit `plugin_root` (or deriving from `binary` by convention). Spike found this on first runtime load.

2. **`wezterm` is NOT a global in mlua** (§9.x): submodules MUST acquire it via `local wezterm = require("wezterm")` at file top, not by reading `_G.wezterm`. The latter resolves to nil and silently breaks runtime detection paths in every submodule that does `if not _G.wezterm then [test mode] end`. Spike's plugin worked around this with one `_G.wezterm = wezterm` assignment in `init.lua` after the require.

3. **`wezterm.GLOBAL` is not a transparent Lua table** (§5.4 / §10.x): nested table values written via `wezterm.GLOBAL.x[id] = {ts = N}` survive the write but throw "can only index array or object values" on subsequent `entry.ts` reads. The spec for `seen_ids`, `wezsesh_state`, and `wezsesh_requests` should either:
   - Use flat integer / string values per ULID (e.g., `seen_ids[ulid] = ts_int` not `{ts = N}`), or
   - Round-trip nested values through `wezterm.json_parse` / `wezterm.json_encode` at the boundary, or
   - Document explicitly that GLOBAL values are limited to JSON-scalar shapes.

   The spike took option (1).

4. **`package.loaded` survives `Ctrl+Shift+R`** (§9.1 / §16): wezterm reloads the user's `wezterm.lua` (re-runs `loadfile`) but does NOT re-evaluate cached `require()` results. Without explicit busting, every submodule edit needs a full wezterm restart to pick up. The spec for `apply_to_config` should mandate either (a) a cache-bust loop over `package.loaded["wezsesh.*"]` keys at entry, or (b) a single source-file plugin layout that doesn't use `require()` for internal modules.

5. **§17.2 fixture id is invalid** — already addressed in PR #4 (id was 27 chars and contained Crockford-excluded glyphs; fixed in `docs/design.md`).

These should all become §0.1 changelog rows + corresponding section edits when the design moves out of v3 draft.

## Spike author's pre-runtime observations

These don't depend on running inside wezterm:

1. **Canonical-JSON encoder is symmetric across Go and Lua for the entire §17.1 fixture corpus** (empty containers, integer extremes, `\\` / `\"` / `\u00XX` controls including U+0080–U+009F, U+2028 / U+2029, multibyte UTF-8 raw, CJK, emoji, forward-slash never escaped, sorted keys). 41 cases verified.

2. **The pure-Lua SHA-256 (`plugin/wezsesh/sha256.lua`, ~80 LOC) matches openssl** for the empty string, `"abc"`, the quick-fox phrase, the 56-byte padding-boundary case, and the 1 MiB `"a"` repetition. The 1 MiB case completes in well under 100 ms in a standalone Lua run (sub-millisecond per typical noop payload).

3. **HMAC-SHA-256 matches openssl** for both the §17.2 fixture and a separate 32-byte-0x0b-key `"Hi There"` vector. The Lua handler's hex-encoded digest equals `crypto/hmac` over the same canonical bytes.

4. **§17.2 fixture issue:** the original `id` literal in design.md was 27 chars and used Crockford-excluded glyphs. The §9.3.1 field-shape validator's `#id == 26` would silent-drop it. Fixed in PR #4.

5. **`wezterm.json_parse` integer subtype:** the design's §9.3.1 validator does NOT explicitly require integer subtype on `target_window_id` / `ts` / `v` (only `type == "number"`). My handler adds `math.type(p.ts) ~= "integer"` and the same for `target_window_id` to pre-empt the §17.5 fuzz mutation class `float_subtype`. Whether `wezterm.json_parse` decodes JSON numbers as Lua integer or float when the value is integer-valued (`1700000000`) is a runtime detail to verify — see scenario 4a (the happy path will fail if json_parse returns floats and we keep the strict check). If runtime shows `wezterm.json_parse` returns floats, document and either relax the validator or normalise to integer.

6. **Constant-time eq edge case:** `ct_eq.eq` returns false for length-mismatched inputs *non-constant-time*. That matches the design's §9.9 stub. The Lua `==` short-circuit only matters for same-length compares, where the byte loop runs to completion regardless of outcome.

7. **The CLI shape** (`wezsesh keygen`, `wezsesh reply <sock> <b64>`, `wezsesh noop`) matches §3.2 / §5.2 / Appendix A. The `noop` subcommand is a spike-only sender; production has it issued by the TUI's bubbletea Update.

8. **SUN_PATH guard** in `buildSockPath` is enforced (104 darwin, 108 linux), with a hard error rather than silent truncation. Worst case observed: `/tmp/wezsesh-501/<8hex>.sock` is 32 chars, far under either limit.

9. **Reply socket is sequential-accept** (cap-2 channel) per §13.2; the spike's `acceptLoop` mirrors that shape but has no started-then-completed split because noop has no started reply.

10. **Retransmit at 2 s** is implemented per §13.1 / §14.1; the second OSC is intended to be silent-dropped by the same `seen_ids` mechanism that the explicit `--replay` test exercises.
